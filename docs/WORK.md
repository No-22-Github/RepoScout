每个功能点都必须遵守下面的顺序：

1. 读功能点定义
2. 建本轮 todo
3. 写代码
4. 写测试或补测试
5. 运行格式化

7. 评估回归影响并运行必要测试
8. 修复代码或修正测试

10. 汇报结果

## 6. 测试与回归规则

本项目默认要求每个 feat 都有自己的验证，不允许“只改代码，不测”。

### 6.1 新功能规则

- 新增功能点时，应尽量增加对应单元测试或最小集成测试
- 如果当前阶段还没有成熟测试基建，至少补一个最小可运行验证

### 6.2 修改已有逻辑规则

- 改动已有逻辑时，必须判断影响范围
- 至少运行与改动直接相关的测试
- 如果改动可能影响公共模块、CLI 入口、配置层、排序逻辑或输出结构，必须扩大测试范围

### 6.3 回归判断规则

如果代码改动可能波及以下内容，必须运行对应回归测试：

- CLI 参数解析
- 配置读取
- JSON 输出结构
- Markdown 输出结构
- 候选集构建
- 排序逻辑
- 模型接口适配
- MCP 服务入口

### 6.4 测试失败时的处理规则

测试失败后，必须先判断是哪一类问题：

1. 代码写错了
2. 旧测试假设不再符合新需求
3. 新需求本身没有表达清楚

处理原则：

- 如果是代码错了，修代码
- 如果是测试过时但新需求是合理的，修改测试
- 如果需求不清楚且继续改有风险，停止并记录阻塞

### 6.5 不允许的行为

- 不运行任何测试就宣称完成
- 只跑自己新加的测试，不管明显受影响的旧测试
- 看到测试失败就直接删测试
- 为了过测试而绕开真实需求

## 12. 对用户的收尾汇报协议

每轮完成后，除了更新文档，还必须向用户做一次规范化汇报。

汇报内容至少包括：

1. 本轮完成的功能点
2. 本轮新增或修改了什么功能
3. 改了哪些关键文件
4. 是否更新了文档
5. 是否写了测试或补了测试
6. 实际运行了哪些格式化和测试
7. 是否满足当前功能点的完成标准
8. 当前残留风险或未完成项
9. 下一个建议功能点

汇报时应明确区分：

- 已完成的事实
- 未执行的验证
- 仍然存在的风险

不允许：

- 没跑测试却写成“已验证”
- 没更新文档却不说明
- 把“应该可以”写成“已经完成”

### 12.1 推荐汇报结构

建议按下面顺序对用户汇报：

- 本轮完成了哪个 `RS-xxx`
- 实现了哪些具体功能
- 更新了哪些文档
- 跑了哪些格式化和测试
- 是否满足完成标准
- 下一步建议做什么

### 12.2 提交前必须询问用户

默认情况下：

- 完成代码、测试、文档更新后，不自动提交
- 必须在收尾汇报最后明确询问用户：`是否需要我直接提交？`

但如果用户在启动当前会话时已经明确要求“完成后直接提交”或给出了等价要求，则以该要求为准，不需要重复询问。

只有用户明确同意后，才可以执行 commit。

### 12.3 用户同意提交后的行为

如果用户明确表示可以提交，则继续执行下面流程：

1. 再次检查本轮改动范围是否和本功能点一致
2. 确认文档状态已经回填
3. 确认至少执行了本轮应做的格式化和测试
4. 生成一条与本轮功能点匹配的 commit message
5. 执行 commit
6. 向用户汇报 commit message

### 12.4 Commit message 规范

commit message 应简洁、准确、和本轮功能点直接对应。

优先风格：

```text
feat(reposcout): <summary>
fix(reposcout): <summary>
docs(reposcout): <summary>
refactor(reposcout): <summary>
test(reposcout): <summary>
```

如果本轮主要对应某个功能点，summary 应尽量体现该功能点内容，例如：

```text
feat(reposcout): scaffold CLI project skeleton
feat(reposcout): add shared runtime config loading
feat(reposcout): define core repo scout schemas
```

### 12.5 不允许的提交行为

- 用户没同意就直接 commit
- commit message 和实际改动不匹配
- 把多个无关功能点硬塞进一次提交
- 测试明显失败时仍然提交，除非用户明确接受当前状态
